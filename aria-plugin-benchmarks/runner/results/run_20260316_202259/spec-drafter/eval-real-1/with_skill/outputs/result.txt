# Tag/Label Feature for Todo Items

> **Level**: Standard (Level 3 Spec)
> **Status**: Draft
> **Module**: frontend, backend
> **PRD Reference**: docs/requirements/prd-todo-optimization-v1.md
> **Created**: 2026-03-16

## Why

Todo items currently have no organizational mechanism beyond completion status. Users working on multiple projects or contexts have no way to categorize, group, or filter tasks semantically. Tags provide a lightweight, flexible taxonomy that requires no upfront structure while supporting powerful filtering workflows.

The database schema already includes a `tags` table and a `todo_tags` join table (confirmed in `backend/src/database.js`), so the infrastructure cost is low — this is primarily a backend API + frontend feature.

## What

Allow users to create reusable tags, assign them to todo items, and filter the todo list by one or more tags.

### FR-TAG-001: Tag Management (P1)

- Create a tag with a name and optional color
- List all tags belonging to the authenticated user
- Delete a tag (removes it from all associated todos)
- Tag names are unique per user, case-insensitive
- Suggested colors: a fixed palette of 8–10 options

### FR-TAG-002: Assign Tags to Todos (P1)

- Assign one or more tags to a todo item (on create or via edit)
- Remove a tag from a todo item
- Display assigned tags as colored chips/pills on each todo card
- Max 5 tags per todo item (UI enforced)

### FR-TAG-003: Filter by Tags (P1)

- Filter bar above the todo list showing all user tags
- Clicking a tag toggles it as an active filter
- Multiple active filters use AND logic (show todos matching all selected tags)
- Active filter state persists in the URL query string (`?tags=work,urgent`)
- "Clear filters" button to reset

### FR-TAG-004: Tag Autocomplete in Todo Form (P2)

- When assigning tags during create/edit, show autocomplete dropdown from existing tags
- Allow creating a new tag inline from the dropdown if name doesn't exist yet

## API Design

```
GET    /api/tags              — list all tags for current user
POST   /api/tags              — create tag { name, color }
DELETE /api/tags/:id          — delete tag

GET    /api/todos             — include tags[] in response (existing endpoint, extend)
POST   /api/todos/:id/tags    — assign tag { tagId } to todo
DELETE /api/todos/:id/tags/:tagId — remove tag from todo
```

## Impact

- **影响范围**: Backend API (new routes) + frontend components; existing todo endpoints extended minimally
- **Database**: `tags` and `todo_tags` tables already exist — no migration needed
- **风险**: Low — additive change; existing todo functionality unaffected
- **Auth**: All tag endpoints require authentication; tags are user-scoped
