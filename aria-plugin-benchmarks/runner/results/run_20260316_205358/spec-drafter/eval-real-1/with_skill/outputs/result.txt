# Phase 5: Todo 标签/标记功能

> **Level**: Standard (Level 2 Spec)
> **Status**: Draft
> **Module**: backend, frontend
> **PRD Reference**: docs/requirements/prd-todo-optimization-v1.md
> **Created**: 2026-03-16

## Why

当前 Todo 项只能通过完成状态和文本内容区分，缺乏分类组织能力。用户在处理多个项目或工作流时，
无法快速识别和筛选特定类型的任务，导致列表混乱、效率降低。

数据库层已预建 `tags` 和 `todo_tags` 表，具备完整的基础设施，只需实现 API 和前端即可交付完整功能。

## What

为 Todo 系统添加标签/标记功能，覆盖三个核心用户流程：

### FR-TAG-001: 标签管理 (P1)

用户可以创建、修改、删除全局标签：

- 创建标签（名称 + 颜色选择）
- 编辑标签名称和颜色
- 删除标签（从所有关联 todo 中移除）
- 标签名称在用户范围内唯一（大小写不敏感）
- 支持颜色：预设 8 种 + 自定义 hex 颜色输入

### FR-TAG-002: 标签分配 (P1)

用户可以为 Todo 项添加/移除标签：

- 在 Todo 创建时选择标签
- 在 Todo 详情/编辑模式中添加或移除标签
- 单个 Todo 最多支持 10 个标签
- 标签以彩色 pill/badge 形式显示在 Todo 列表行内

### FR-TAG-003: 按标签筛选 (P1)

用户可以通过标签过滤 Todo 列表：

- 侧边栏或标签栏显示所有标签及其关联的 todo 数量
- 点击标签进行单标签筛选
- 支持多标签筛选（AND 逻辑：仅显示包含所有选中标签的 todo）
- 筛选状态通过 URL query string 持久化（`?tags=1,3`）
- 筛选与现有的完成状态筛选可组合使用

## Key Deliverables

| 交付物 | 类型 | 说明 |
|--------|------|------|
| 标签 CRUD API | 新增 | `GET/POST /api/tags`, `PUT/DELETE /api/tags/:id` |
| Todo-标签关联 API | 新增 | `POST/DELETE /api/todos/:id/tags/:tagId` |
| 带标签筛选的 Todo 列表 API | 修改 | `GET /api/todos?tags=1,3` 支持多标签过滤 |
| 标签选择器组件 | 新增 | `frontend/src/js/components/TagPicker.js` |
| 标签 Badge 组件 | 新增 | `frontend/src/js/components/TagBadge.js` |
| 标签管理界面 | 新增 | 设置页或侧边栏标签列表 |
| 标签筛选 UI | 新增 | 标签栏/侧边栏 + URL 状态同步 |
| 标签样式 | 新增 | `frontend/src/css/tags.css` |

## API Design

```
# 标签 CRUD
GET    /api/tags              # 获取当前用户所有标签（含 todo 数量）
POST   /api/tags              # 创建标签 { name, color }
PUT    /api/tags/:id          # 更新标签 { name?, color? }
DELETE /api/tags/:id          # 删除标签

# Todo 标签关联
GET    /api/todos/:id/tags    # 获取 todo 的标签列表
POST   /api/todos/:id/tags    # 分配标签 { tagId }
DELETE /api/todos/:id/tags/:tagId  # 移除标签

# Todo 列表（扩展现有接口）
GET    /api/todos?tags=1,3    # 按标签 ID 过滤（AND 逻辑）
```

## Impact

- **影响范围**: 后端新增路由和服务层 + 前端新增组件和样式
- **数据库**: 无需迁移 — `tags` 和 `todo_tags` 表已存在
- **风险**: 低 — 新功能不影响现有 todo CRUD 路径，筛选 API 向后兼容
- **认证**: 标签需绑定 `user_id`（`tags` 表当前缺少此字段，需添加列）

## Schema Gap

`tags` 表当前无 `user_id` 字段，需在初始化脚本中添加（或通过迁移）：

```sql
ALTER TABLE tags ADD COLUMN user_id TEXT;
CREATE INDEX idx_tags_user_id ON tags(user_id);
```

---

**Key design decisions:**

1. **No DB migration needed for junction table** — `todo_tags` already exists with proper FK cascades.
2. **One schema gap** — `tags` table needs `user_id` for multi-user isolation (currently missing).
3. **AND logic for multi-tag filter** — stricter but more useful than OR; matches "show me tasks that are both `urgent` AND `work`".
4. **URL persistence** — `?tags=1,3` makes filtered views shareable/bookmarkable.
