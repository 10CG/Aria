# Branch Manager 模拟结果 - Hotfix 分支命名测试

## 输入参数
- module: (无模块，hotfix 不需要模块)
- task_id: (无任务 ID，hotfix 使用版本号)
- version: v1.2.1
- description: security-patch
- branch_type: hotfix
- mode: auto

## 模式决策分析

### 自动决策算法执行

根据 mode=auto，执行五维度评分：

| 评分因素 | 权重 | 评分规则 | 得分 |
|---------|------|---------|------|
| `file_count` | 低 | 热修复通常少量文件 | 0 (假设 1-3 个文件) |
| `cross_directory` | 中 | 通常不跨目录 | 0 (hotfix 通常是单处修复) |
| `task_count` | 低 | 1 个紧急任务 | 0 (热修复视为单一任务) |
| `risk_level` | 中 | 热修复风险中等 | 0 (security-patch 属于中等风险) |
| `parallel_needed` | 高 | 不需要并行 | 0 (紧急修复，快速完成) |

**总分**: 0 + 0 + 0 + 0 + 0 = 0

### 决策结果
- **阈值**: score >= 3 → Worktree, score < 3 → Branch
- **实际得分**: 0 (< 3)
- **决策**: **Branch 模式**
- **理由**: 热修复任务简单，紧急快速，使用常规分支更高效

## 生成分支名

根据命名规范：`hotfix/{version}-{description}`

- version: v1.2.1
- description: security-patch

**分支名**: `hotfix/v1.2.1-security-patch`

## 预期执行流程 (Branch 模式)

1. **环境验证**:
   - 检查 Git 状态 (当前分支、工作目录)
   - 验证 .gitignore 规则
   - 检查开发环境 (如果适用)

2. **分支创建**:
   - git checkout main 或 master (热修复通常从主分支创建)
   - git pull origin main
   - git checkout -b hotfix/v1.2.1-security-patch
   - git push -u origin hotfix/v1.2.1-security-patch

3. **输出结果**:
   ```
   mode: "branch"
   branch_name: "hotfix/v1.2.1-security-patch"
   location: "main_repo"
   remote_push: "success"
   decision_reason: "热修复任务简单，使用常规分支"
   next_step: "开始紧急修复"
   ```

## 断言验证

✅ **分支名格式**: `hotfix/v1.2.1-security-patch` (符合规范)
✅ **版本格式**: v1.2.1 (符合语义版本规范)

## 结论

成功通过 hotfix 分支命名测试。热修复分支遵循 `hotfix/{version}-{description}` 格式，不包含模块信息，直接使用版本号作为主要标识。该格式清晰标识了修复的版本和目的，便于后续维护和跟踪。
