## 1. Executive Summary

| Sub-test | What it measures | Result | Threshold | Verdict |
|----------|------------------|--------|-----------|---------|
| **T3.4.a** (auto) | GLM-4.7 自主从自然语言决定调用 state-scanner Skill | 0/5 PASS | ≥4/5 | **FAIL** |
| **T3.4.b** (slash, original grep `Phase/Cycle:\|phase_cycle:`) | 显式 `/state-scanner` → Skill 执行 → UPM 字段输出 | 0/5 PASS | ≥4/5 | **FAIL (spec grep bug)** |
| **T3.4.b** (slash, corrected grep) | 显式 `/state-scanner` → Skill 执行 → canonical 格式 | 3/5 PASS | ≥4/5 | FAIL (3 < 4, 但 2 个失败是 timeout 非 Skill 问题) |

**核心发现 (Material)**:
1. **GLM-4.7 不自主调用 Claude Code Skills** — 即使 `bypassPermissions`, GLM 倾向于从 context/memory 直接 narrate, 不走 Skill 框架 (T3.4.a 0/5)
2. **Slash 命令可靠触发 Skill** — 当 GLM 收到显式 `/state-scanner` 输入, 会忠实执行 Skill, 产出 canonical 格式 (T3.4.b 3/3 non-timeout PASS)
3. **Luxeno + claude-CLI 存在上游可靠性问题** — 2/5 slash 样本 120s timeout (其他场景也见偶发 408 upstream error)
4. **T3.4 spec grep 模式是项目耦合 bug** — `Phase/Cycle:|phase_cycle:` 仅在 UPM-enabled 项目中出现, Aria 作为 meta-project 不走 UPM, state-scanner 自动省略该字段; T3.5 AI 预审未捕获 (UPM 假设)

## 2. Testing Scope & Method

### Routing 验证

| 链路 | 端点 | 模型 | 结果 |
|------|------|------|------|
| Prompt 生成 (GLM humanize template) | Luxeno `/v1/chat/completions` (OpenAI-compat) | `glm-4.5` | ✅ Reliable (4/5, 1 次 reasoning token 耗尽) |
| claude -p (plugin + Skill) | Luxeno `/v1/messages` (Anthropic-compat) | `glm-4.7` | ⚠️ 偶发 timeout + 偶发空 stdout |

**Spec vs 实测偏差**:
- Spec 要求 `glm-4.7-flashx`, 但:
  - OpenAI endpoint 上 flashx 在 ≥800 byte 输入上返回 408 upstream error (Luxeno 侧)
  - Anthropic endpoint 上 flashx "not available", 只接受 `glm-4.7` (flagship)
- 实测妥协: OpenAI endpoint 用 glm-4.5, Anthropic endpoint 用 glm-4.7

### 执行参数

- `claude -p --permission-mode bypassPermissions <prompt>`
  - `bypassPermissions` 必需, 否则 GLM 会 ask-for-approval 然后挂起 (headless 模式收不到 y/n 响应)
- `ANTHROPIC_BASE_URL=https://api.luxeno.ai`
- `ANTHROPIC_MODEL=glm-4.7`
- Timeout: 120s per call

### Sample set

5 个 Jinja2 模板 (T3.5 approved), seeds `a1`/`b2`/`c3`/`d4`/`e5`:
- `01-initial-scan` 初次打开项目
- `02-resume-after-break` 暂别归来
- `03-pre-action-guidance` 有变更想决策
- `04-ambiguous-request` 模糊请求
- `05-anomaly-triage` 自查异常

## 3. T3.4.a 详细结果 (auto mode, 0/5)

| # | Template | GLM prompt (生成的自然语言) | claude -p 结果 | failure_mode |
|---|----------|------------------------------|----------------|--------------|
| 1 | initial-scan | "刚切换到这个分支，完全忘了上次在哪了..." | 583 chars, narrative | `triggered_invalid_yaml` |
| 2 | resume-after-break | "嘿，我几天没碰这个项目了..." | 1074 chars, narrative | `triggered_invalid_yaml` |
| 3 | pre-action-guidance | "刚改了几个文件还没提交..." | 553 chars, narrative | `partial_response` |
| 4 | ambiguous-request | (GLM reasoning 耗尽 max_tokens, 重试 2 次全空) | N/A | `glm_generation_failed` |
| 5 | anomaly-triage | "我感觉这个仓库状态有点飘..." | timeout (120s) | `timeout` |

**Pattern**: GLM-4.7 收到自然语言, 选择"基于已知 context 直接回答" 而非"调用 state-scanner Skill"。即使 bypassPermissions 已消除审批门槛, GLM 仍不自主发起 Skill 调用。

**A1.c 结论** (PRD §A1 的 "自然语言 → Skill 自动执行" 子假设): **不成立** (在 GLM-4.7 via Luxeno 配置下)。

## 4. T3.4.b 详细结果 (slash mode, 3/5 corrected)

| # | Template | claude -p 结果 | Original grep | Corrected grep | Verdict |
|---|----------|----------------|---------------|----------------|---------|
| 1 | initial-scan | timeout (120s, 0 stdout) | FAIL | FAIL | timeout |
| 2 | resume-after-break | 2046 chars, `PROJECT STATE ANALYSIS` 格式 | FAIL (no Phase/Cycle) | **PASS** | Skill ✅ |
| 3 | pre-action-guidance | 2347 chars, `PROJECT STATE ANALYSIS` 格式 | FAIL (no Phase/Cycle) | **PASS** | Skill ✅ |
| 4 | ambiguous-request | timeout (120s, 0 stdout) | FAIL | FAIL | timeout |
| 5 | anomaly-triage | 2649 chars, `PROJECT STATE ANALYSIS` 格式 | FAIL (no Phase/Cycle) | **PASS** | Skill ✅ |

**Pattern**: 3 个 non-timeout 样本 **全部成功调用 state-scanner Skill 并产出 canonical 格式**, 证明:
- ✅ aria-plugin 在 headless `claude -p` 下可加载 (A1.b 成立)
- ✅ Skill routing 经 Luxeno→glm-4.7 可用 (GLM 遵循 `/slash-command` 约定)
- ⚠️ 可靠性问题: 40% timeout 率 (Luxeno Anthropic endpoint 有上游 hangs)

**Grep 修正**:
- Original: `Phase/Cycle:|phase_cycle:` — 仅 UPM 项目输出
- Corrected: `PROJECT STATE ANALYSIS|📍 当前状态|🎯 推荐工作流` — state-scanner canonical 标记, 项目无关

**T3.5 AI 预审 gap**: ai-engineer 签字了这份 grep, 但未识别"Aria 作为 meta-project 不走 UPM, Phase/Cycle 永远空"。T3.5 Finding #1 引入的双格式容忍 (`Phase/Cycle:|phase_cycle:`) 治标不治本。

## 5. 对 AD3 (Hermes Option C Extension-only) 的影响

**强化决议正确性**:
- AD3 反转 (fork → Option C) 的核心逻辑是 "Layer 2 用 Extension tool pack, 不依赖 LLM 推理 Skill"
- T3.4.a 证实 GLM-4.7 不擅长 "从自然语言推理应调用哪个 Skill"
- T3.4.b 证实 GLM-4.7 遵循显式调用契约 (`/skill-name` 或 extension tool API)
- → **Option C 的显式 API 契约 = 与 GLM 能力边界 match**

若当初选择 fork 路径 (依赖 LLM 自主调 Skills), 本轮 T3.4 发现会是灾难性的 Layer 2 设计暴雷。M0 Spike (2026-04-16, 4.5h POC 13/13) + 本次 T3.4 findings 形成双重验证。

## 9. 反思 — 本轮 T3.4 执行的 5 个教训

1. **Luxeno endpoint routing 需要实测发现, 文档不全**: OpenAI `/v1` vs Anthropic `/v1/messages` + 不同模型可用性, 没有一处 docs 说清, 只能 probe (消耗 0.5h)
2. **Spec 里写的 `glm-4.7-flashx` 是假设而非验证**: T3.5 AI 预审没有实测, 只做文本审查; 应增加一步 "config smoke" 验证 (model alive) 再正式批准 fixture
3. **Grep criterion 应用 canonical 标记而非项目字段**: 跨项目测试的判定器要用"框架级稳定标记" (emoji header, 格式框) 而非"项目级可选字段"
4. **Timeout rate 40% 是 M0 需记录的 infra 基线**: 不阻塞 Go 决议, 但 M1 Layer 1 状态机需把 Luxeno retry/fallback 作为一等公民
5. **T3.4 拆成 a+b 的价值**: 比单一 "pass/fail" 承载更多信息密度, 为 Option C 提供正向证据, 为自然语言 auto-trigger 提供反向证据; 建议未来 Aria spike-style 测试都采用"分层判定"原则
