## Security Analysis

### [CRITICAL] SQL Injection Vulnerability

**File**: `backend/services/user_service.py`, line 52
**Issue**: Raw string interpolation in SQL query - SQL injection risk

```python
# VULNERABLE CODE:
query = f"SELECT * FROM users WHERE email = '{email}'"
db.execute(query)

# SECURE FIX (use parameterized query):
query = "SELECT * FROM users WHERE email = %s"
db.execute(query, (email,))
```

**Risk**: An attacker can inject malicious SQL by manipulating the `email` input.

### [CRITICAL] Hardcoded Secret Key

**File**: `backend/utils/jwt_handler.py`, line 8
**Issue**: JWT secret key hardcoded in source code

```python
# VULNERABLE CODE:
SECRET_KEY = "my-secret-key-12345"

# SECURE FIX (use environment variable):
SECRET_KEY = os.environ.get("JWT_SECRET_KEY")
if not SECRET_KEY:
    raise EnvironmentError("JWT_SECRET_KEY environment variable not set")
```

**Risk**: Secret key exposed in version control. Any JWT token can be forged.

## Code Style Review

### [MINOR] Naming Convention Inconsistency

**File**: `backend/api/auth/login.py`
**Issue**: Mixed naming styles detected

```python
# Inconsistent (snake_case mixed with camelCase):
userID = user.id          # should be user_id
loginTime = datetime.now() # should be login_time
```

Follow PEP 8: use `snake_case` for all Python variables.

## Architecture Assessment

### [IMPORTANT] Missing Error Handling Layer

**Pattern**: No centralized error handling for database failures

The current architecture handles errors locally in each endpoint, which leads to:
- Inconsistent error responses across endpoints
- Duplicate error handling code
- Missing logging for infrastructure failures

**Recommended Design Pattern**: Add a global exception handler

```python
@app.errorhandler(Exception)
def handle_exception(e):
    logger.error(f"Unhandled exception: {e}", exc_info=True)
    return jsonify({"error": "Internal server error"}), 500
```
