# T2.4 — Nomad Meta 边界实测 + R7 缓解方案

> **Spec**: [aria-2.0-m0-prerequisite](../../proposal.md) §T2.4
> **执行**: 2026-04-15
> **结论**: ✅ PRD R7 的 "64KB" 假设**偏保守**。Nomad meta **单 key value 实际上限 ≈ 131048 字节 (128 KiB − 24 字节 env 前缀)**, 由 Linux 内核 `MAX_ARG_STRLEN = 131072` 决定。R7 缓解方案仍然正确 (prompt 走文件, meta 传 ISSUE_ID), 但原因从 "Nomad 有 64KB 限制" 更正为 "Linux execve 单字符串 128KB 限制"。

---

## TL;DR

| 问题 | PRD R7 原陈述 | 实测结论 |
|------|--------------|---------|
| 限制层 | "Nomad meta 64KB" | 实际是 **Linux 内核 `MAX_ARG_STRLEN`**, 不是 Nomad |
| 限制值 | 64 KB | **131048 字节 per key** (key 名 "target_node" 11 字符时), 通用公式 `max_value = 131059 - len(key)` |
| 失败模式 | (未描述) | 容器 `execve("/bin/sh")` 返回 `E2BIG`, stderr: `exec /bin/sh: argument list too long` |
| 缓解方案 | "prompt 走文件, meta 传 ISSUE_ID" | **方案本身保持不变**, 但阈值从 "64KB" 改为 "100KB per field" (保守线) |

## 1. 实验方法

复用 T2.2 的 `aria-storage-smoke.hcl` (parameterized batch job), 通过未被脚本使用的 `meta_optional = ["target_node"]` 字段作为可变大小载体:

```python
meta = {'smoke_id': f'probe-{label}', 'target_node': 'x' * N}
POST /v1/job/aria-storage-smoke/dispatch {'Meta': meta}
```

- `target_node` 只作为 env var 注入容器, 不被 task 脚本引用
- 逐步扩大 N, 观察 (a) dispatch HTTP API 是否 accept, (b) alloc 是否 complete
- 失败 case 取 alloc stderr 分诊

- 边界落在 100KB ~ 500KB 之间

## 3. 失败根因 — Linux E2BIG, 非 Nomad

取 500K 和 1M 两个失败 alloc 的 stderr:

```
exec /bin/sh: argument list too long
```

这是 Linux kernel 的 `execve(2)` 系统调用返回 `-E2BIG`。内核源码 (`fs/exec.c`):

```c
#define MAX_ARG_STRLEN (PAGE_SIZE * 32)  // 131072 on 4K page

static int copy_string(int argc, const char __user *const __user *argv, ...) {
    ...
    if (len == 0 || len > MAX_ARG_STRLEN)
        return -E2BIG;
    ...
}
```

**含义**: execve 传入的**每一个**单独的 argv/envp 字符串 (含 NUL 终止符) 不得超过 `MAX_ARG_STRLEN = 32 * PAGE_SIZE = 131072 字节`。

**注意**: 这是**单字符串**上限, 不是 argv+envp 总和。总和另有更大限制 (`MAX_ARG_PAGES = 1/4 * RLIMIT_STACK`, 通常 ≥ 2MB)。所以多个中等大小的 env var 可以并存, 但任何**单个** env var 字符串不得超过 128 KiB。

## 4. 精确边界二分

围绕 `MAX_ARG_STRLEN = 131072` 二分:

| Label  | Value bytes | Total env string bytes<sup>*</sup> | Dispatch | Alloc |
|--------|-------------|------------------------------------|----------|-------|
| 120K   | 122880 | 122903 | ok | **complete** |
| 128K   | 131072 | 131095 | ok | **failed** |
| 131K   | 134144 | 134167 | ok | failed |
| 131040 | 131040 | 131063 | ok | **complete** |
| 131048 | 131048 | **131071** | ok | **complete** |  ← **最后成功**
| 131049 | 131049 | **131072** | ok | **failed**   |  ← **第一失败**
| 131050 | 131050 | 131073 | ok | failed |
| 131072 | 131072 | 131095 | ok | failed |
| 150K   | 153600 | 153623 | ok | failed |

<sup>*</sup> `NOMAD_META_target_node=<value>` = 23 chars prefix + value + 1 NUL terminator

### 理论与实测对应

Linux kernel 判定:
```
fail if (len > MAX_ARG_STRLEN)    // 即 len > 131072
```

其中 `len` 是 `strnlen_user()` 返回的字符串长度**包含 NUL**。

- value = 131048 → env string total (incl NUL) = 23 + 131048 + 1 = **131072** ✅ (恰好等于上限, 不 > 上限, 通过)
- value = 131049 → env string total = 23 + 131049 + 1 = **131073** ❌ (> 131072, E2BIG)

**边界公式**:
```
max_value_bytes = MAX_ARG_STRLEN - len("NOMAD_META_") - len(key) - len("=") - 1 (NUL)
                = 131072 - 11 - len(key) - 1 - 1
                = 131059 - len(key)
```

对于本实验中的 key `target_node` (11 字符): `max = 131059 - 11 = 131048` ✓ 与实测一致。

**参考值**:
| key 名 | max value bytes |
|-------|-----------------|
| `p` (1) | 131058 |
| `issue_id` (8) | 131051 |
| `target_node` (11) | **131048** (本实验实测) |
| `prompt_text` (11) | 131048 |
| `hermes_dispatch_payload` (23) | 131036 |

## 5. 对 PRD R7 的结论

### R7 原表述 (从 Spec 引用)

> Nomad meta 64KB 边界测试 (0.5h)
> - 发送一个 60KB 的 meta 参数 dispatch
> - 确认边界行为, 产出 R7 缓解方案文档 (prompt 走文件, meta 传 ISSUE_ID)

PRD v2.0 §风险章节应有对应条目 (未现场查证):

> R7: Nomad meta 字段限制 ~64KB, 拟人命令 prompt 可能超限 → 缓解: prompt 文件化 + meta 传引用

### 修正后的陈述

1. **限制层更正**: 不是 Nomad 限制, 是 **Linux execve 的 `MAX_ARG_STRLEN` per-string 限制** (128 KiB)
2. **数值修正**: 单 meta value 上限 **~128 KiB**, 不是 64 KiB; 实测每 key 131048 bytes (key 名 11 字符时)
3. **适用环境**: 此限制存在于所有 Linux 容器运行时 (docker / containerd / raw exec), 与 Nomad 版本无关
4. **绕过可能**: 不可绕过 (不改内核), 除非:
   - 使用多个较小的 meta key (总和上限由 `MAX_ARG_PAGES` ≈ 2 MB+ 决定)
   - 通过 payload 传递 (payload 是 stdin, 不经 execve argv, 无 E2BIG)
   - 通过文件系统传递

### R7 缓解方案 (修订版)

**保持不变的核心建议**:
- ✅ prompt 文本**永远**通过共享存储文件传递 (`/opt/aether-volumes/aria-runner/inputs/<trace_id>.prompt`)
- ✅ meta 只承载**小的结构化标识符** (trace_id, issue_id, priority, model 等)

**新增的设计约束**:
1. **单 meta key 字节预算**: ≤ **100 KB** (留 28 KB 缓冲给 key 名 + 内核 slop)
2. **多 meta key 总预算**: ≤ **500 KB** (远低于 `MAX_ARG_PAGES` 硬限, 留充足安全边界)
3. **任何结构化载荷 ≥ 1 KB**: 优先 payload (stdin) 或文件, 避免堆积 meta
4. **Layer 1 → Layer 2 拟人命令**: 命令本体走文件, meta 只传:
   ```
   trace_id=<uuid>
   issue_id=<forgejo_issue_number>
   command_file=/opt/aria-outputs/<trace_id>/command.md
   prompt_file=/opt/aria-inputs/<trace_id>/prompt.txt
   ```

### Hermes (Layer 1) 代码侧约束

```python
# pseudo
def dispatch_runner(trace_id, issue_id, prompt_text):
    input_dir = f'/opt/aether-volumes/aria-runner/inputs/{trace_id}'
    os.makedirs(input_dir, exist_ok=True)
    with open(f'{input_dir}/prompt.txt', 'w') as f:
        f.write(prompt_text)  # 无大小限制 (磁盘空间决定)
    
    nomad_dispatch('aria-runner-claude', meta={
        'trace_id': trace_id,          # ~36 bytes
        'issue_id': str(issue_id),     # ~10 bytes
        'prompt_file': f'{input_dir}/prompt.txt',  # ~100 bytes
    })
```

### 可观测性要求

- Hermes 在 dispatch 前必须对每个 meta value 做长度断言 `assert len(v) < 100 * 1024`
- aria-runner 启动脚本必须 fail-fast 检查 `NOMAD_META_prompt_file` 是否存在且可读
- 两端都把字节长度记入 trace log, 便于 post-mortem

## 6. 顺带发现 (非 T2.4 范围但值得记录)

### 6.1 空 value 行为

实验未测 value='' (0 字节)。根据内核代码 `if (len == 0 || len > MAX_ARG_STRLEN)` , 0 字节理论也会 E2BIG。但 Nomad 可能在注入 env 前做非空检查, 实际行为待测。**不影响 R7 结论**。

### 6.2 dispatch API 的宽松性

Nomad `/v1/job/.../dispatch` API 对 Meta 大小零校验 (1MB 请求 200 OK)。错误**全部**推迟到 task 启动时的容器 exec 阶段。含义:
- **不能**依赖 dispatch 成功推断 alloc 会成功
- Hermes 必须在**客户端**做大小检查, 而不是等 Nomad 报错

### 6.3 失败是 silent 的

失败 alloc 的 stderr 只有一行 `exec /bin/sh: argument list too long`, 没有任何 meta/env 相关的提示。生产环境 debug 时很容易误判为 "脚本问题" 而非 "参数过大"。建议 Hermes 的错误分诊 rule 专门识别此错误字符串。
