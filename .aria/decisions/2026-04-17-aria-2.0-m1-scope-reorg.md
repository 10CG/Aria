# 决策: Aria 2.0 M1/M2 Story 范围重排 (2026-04-17)

> **Status**: Approved (per AD-M0-9 solo-lab, owner 决议)
> **Triggered by**: M0 Go-with-revision (2026-04-17), US-021 post_spec 收敛审计 Round 1 KM-C1 finding
> **Affects**: PRD v2.1 §User Stories 表格 + US-021 / US-022 scope 定义

## 背景

PRD v2.1 §User Stories 表格 (line 503-514) 原按 "架构层/功能域" 分组:
- US-020 = Aria 2.0 两层架构基础设施
- US-021 = Layer 1 状态机 + Hermes Option C Extension
- US-022 = Layer 2 容器镜像 + 拟人命令协议

M0 (US-020) 实际执行后, US-020 被重定为 "M0 前置验证与架构定稿" (Level 3 Spec done 2026-04-17)。PRD 表格未随 v2.1 patch 同步, 造成 stale 数据。

post_spec Round 1 audit (2026-04-17) knowledge-manager agent 发现命名冲突: **PRD 表格定义 US-021 = Layer 1 状态机, 但 M0 Report §6 定义 US-021 = M1 MVP**, 两者倒置。新读者读 PRD 表格会得到与实际 Spec 完全相反的 Story 范围理解。

## 决策

**范围重排**: US-021~027 改按 "milestone 时间序" 分组, Story 与 M1-M6 一一对应。

| ID | 原 (PRD 表格 stale) | 新 (按 milestone 时间序) |
|----|---------------------|-------------------------|
| US-020 | Aria 2.0 两层架构基础设施 | **Aria 2.0 M0 前置验证与架构定稿** (done 2026-04-17) |
| US-021 | Layer 1 状态机 + Hermes | **Aria 2.0 M1 — MVP 手动 E2E dispatch (Layer 2 容器 + Nomad)** |
| US-022 | Layer 2 容器镜像 | **Aria 2.0 M2 — Layer 1 状态机 + Hermes Option C Extension** |
| US-023 | 跨 tick 长任务 + Crash recovery | **Aria 2.0 M3 — 双 provider + Nomad integration + Crash recovery** |
| US-024~027 | 不变 | 不变 (M4-M6 + cross-milestone) |

## 理由

1. **PRD 表格的使用方式**: 是新参与者 (包括 AI agent 和人类 onboarding) 查找 Story 范围的第一入口。stale 数据会直接导致 scope 误解。
2. **milestone 时间序对齐**: Aria 十步循环的 A.0 state-scanner + B.1 branch 都按 milestone 粒度运转, Story 与 milestone 1:1 映射使 workflow 更直觉。
3. **与 M0 Report §6 对齐**: M0 Report 已明确 US-021 = M1 MVP, US-022 = Layer 1 状态机。本决策是把 PRD 表格对齐 M0 Report 的既成事实, 而非新决策。
4. **AD3 反转影响**: Option C Extension-only 决议把原 "Hermes fork" 工作量大幅下降 (52h → 4.5h), 使 Layer 1 集成推迟到 M2 合理可行; 这进一步强化了 "M1 只做 Layer 2 MVP, M2 做 Layer 1 状态机" 的切分。
