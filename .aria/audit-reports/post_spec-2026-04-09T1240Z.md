### 🔴 MAJOR Issues (10)

#### M1. 伪依赖: Spec#2 依赖 Spec#1 的 fail-soft 框架实际不存在
- **Found by**: tech-lead, code-reviewer (partial)
- **Scope**: `issue-awareness/proposal.md` — "Depends On" 字段 + D1
- **Problem**: Spec#1 仅描述自身 fail-soft 行为，未抽象任何可复用的 lib/schema/helper
- **Fix**: 将 "Depends On" 改为 "Related To"，解耦两 Spec；或在 Spec#1 显式定义 fail-soft pattern 作为共享契约

#### M2. 双版本发布缺乏用户价值论证
- **Found by**: tech-lead
- **Scope**: 两 Spec 的 Target Version
- **Problem**: 两周内 v2.8→2.9→2.10 给 plugin 用户制造升级噪音；AB delta 归因可用 feature flag 替代
- **Fix**: 合并为 v2.9.0 单次发布（sync_check 默认 on + issue_scan 默认 off），或在 D10 补充"独立发布的用户价值"（如紧迫痛点）

#### M3. Spec#1 未覆盖 upstream 未配置场景
- **Found by**: backend-architect
- **Scope**: `sync-check/proposal.md` — ahead/behind 计算
- **Problem**: `git rev-list --count <upstream>...HEAD` 在 upstream 未 set-upstream-to 时直接 exit 1，不会静默
- **Fix**: 先用 `git rev-parse --abbrev-ref @{u}` 探测 upstream，缺失则 `ahead/behind: null` + 原因标注

#### M4. Submodule `remote_commit` 来源不可靠
- **Found by**: backend-architect
- **Scope**: `sync-check/proposal.md` — `sync_status.submodules[].remote_commit`
- **Problem**: `origin/HEAD` 在绝大多数 Aria 子仓库中不存在（需显式 `git remote set-head`），实现者遭遇静默错误
- **Fix**: 明确 fallback 链：`origin/HEAD` → `git ls-remote origin HEAD` → 读取 `.git/config` 中的 default branch → null

#### M5. Spec#2 平台检测的 config schema 字段未扩展
- **Found by**: backend-architect, code-reviewer (T0)
- **Scope**: `issue-awareness/proposal.md` — 平台检测优先级 + config-loader
- **Problem**: Spec 要求读 `.aria/config.json` 的 `platform` 字段，但 `config.template.json` 无此字段；T1/T9 未覆盖 schema 扩展
- **Fix**: 增加 **T0** 任务 (0.25h)：扩展 config schema + 更新 template 文件 + config-loader 默认值

#### M6. 阶段数膨胀 (13 子阶段) 未设上限

- **Scope**: `sync-check + issue-awareness` 合计
- **Problem**: 11→13 接近线性认知极限，Spec 未回答"是否合并语义相近阶段"
- **Fix**: 两 Spec 择一增加 Decision Record: "阶段数上限 15，超出需重构为分组"

#### M7. AC 字段语义不一致

#### M9. T3 错误场景与 proposal 枚举不对齐
