## User Story

_TBD — M1 Go 决议后填充 (owner: solo-lab)_

**草稿方向** (待 M1 handoff 后由 task-planner 细化):
> **作为** Aria 2.0 方法论实施者,
> **我希望** 在 US-021 M1 Layer 2 基础上集成 Layer 1 Hermes Option C Extension + SQLite 状态机, 让系统从 "手动 dispatch" 升级为 "cron 自动 tick + 跨 tick polling",
> **以便** 证明 Aria 2.0 完整两层架构可工作, 为 M3 crash recovery + 双 provider 集成提供稳定状态机基座.

## Scope 摘要

把 US-021 M1 MVP 的"手动 dispatch"升级为"状态机驱动的自动 dispatch", 引入 Layer 1 Hermes Option C Extension 作为长期运行的 orchestrator。

**核心交付** (per M0 Report §6.4 预期):
- Hermes Extension plugin 实现 (Option C, 0 hermes core 修改, 0 rebase 成本) — 引用 M0 Spike ST3 POC 13/13 tests
- SQLite 持久化层 (`dispatches.db`, WAL mode, persistent volume)
- 9 状态机 + S_FAIL 兜底转换逻辑
- 60min cron tick + 跨 tick polling
- Layer 1 → Layer 2 ISSUE_ID 协议 (per R7 Nomad meta 128 KiB, prompt 走 bind mount)
- Feishu webhook (S7 人类 gate) 接入 (M4 完整实现前的桩)

**估算**: ~140h / 6 weeks (Week 7-12, per PRD v2.1 §里程碑概览)

**M1 input 契约** (从 [m1-handoff.yaml schema v1.0](../../../openspec/changes/aria-2.0-m1-mvp/proposal.md) 消费):
- `registry_url` + `image_refs.{immutable_sha, mutable_tag, production_pin}` + `image_tags` (legacy 数组) — Layer 2 镜像契约 (只读, additive-only)
  - **M2 生产部署消费规则 (per AD-M1-2 / TL-R3-4)**: 必须 pin `image_refs.immutable_sha` (content-addressed), **禁引用 `mutable_tag` (claude-latest) 作为 Nomad job image** (M2 开发期 M1 dev 可能刷新覆盖, 破坏可复现性)
- `nomad_job_id` + `nomad_job_version` + `host_volumes` — Nomad deployment 契约
- `performance_baseline.demo_002_p50_duration_s` — non-regression 基线 (M2 集成后 ≤ baseline × 1.5)
- `demo_token_usage` — 成本基线
- `legal_assumptions.anthropic_api_terms_verified` — 合规前提
