## User Story

**作为** 多子模块 Aria 项目的维护者 (以及 Aria 方法论的其他使用者)，
**我希望** 在 state-scanner 扫描项目状态时自动感知本地与远程的同步情况和 Git 平台的 open issues，
**以便** 不会在过时代码上做错误的工作流推荐，也不会错过阻塞性 Issue。

## 背景

state-scanner v2.8.0 已有 11 个子阶段覆盖 Git/OpenSpec/需求/架构/审计/自定义检查，唯独缺失：

1. **本地与远程的同步状态** — 导致多会话/多 agent 并行时频繁出现子模块指针偏移和本地落后远程的误判
2. **Open Issues 感知** — 用户需手动轮询平台检查 "还有什么要做"

## 技术备注

### 双 OpenSpec 并行发布策略

本特性拆分为两个独立但同版本的 OpenSpec:
- **state-scanner-remote-sync-check** (Level 2, 优先) — Phase 1.12
- **state-scanner-issue-awareness** (Level 3, 次期) — Phase 1.13

两 Spec 在 aria-plugin **v1.11.0** 同版本并行发布，无代码依赖，仅术语锚定 (fail-soft 定义)。
合并顺序: aria-plugin#4 先 → Aria#7 后 (主 repo 引用 aria submodule 的新 HEAD)。

### 关键方向性守卫 (pre_merge Round 1 qa_M1 fix)

发现的数据安全级缺陷: 原 `submodule_drift` 规则仅检查 `tree_vs_remote==true` 而不检查方向，
当本地 submodule 领先远程时 (真实发生于 aria 和 aria-orchestrator 子模块)，会错误发出
`git submodule update --remote` 提示，导致本地未推送的 commits 被丢弃。

修复: 规则条件升级为 `tree_vs_remote==true AND behind_count > 0`，新增 `ahead_count` 字段和
`hint_type` ("update" | "push" | "manual_check" | null) 三分支互斥语义。

### 子阶段数量规约 (D8)

state-scanner 子阶段数量从 v2.8.0 的 11 增至 v2.9.0 的 13。D8 引入上限 15 规约，
并明确 **append-only** 阶段编号分配策略 (新阶段一律追加到最大编号 + 1，不允许插入或填补空位)。
