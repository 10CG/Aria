# US-010: Agent Team 质量强化 — Description 精确化 + 协作契约

> **Priority**: HIGH
> **Status**: ready
> **Parent PRD**: prd-aria-v1.md
> **Target Version**: v1.12.0
> **Source**: Forgejo aria-plugin#3 (Layer 1 拆分)

---

## User Story

**作为** 使用 Aria 十步循环的开发者 (及 AI Agent),
**我希望** Agent Team 的 11 个 Agent 有精确的职责边界、明确的触发条件和结构化的协作契约,
**以便** agent-router 路由准确率提升, Agent 间协作高效, 避免 Agent Team 成为十步循环的能力短板。

## 背景

2026-04-11 Agent Team 集体讨论 (aria-plugin#3) 发现:

1. **11 个 Agent 中 6 个 description 路由信号弱** (AI Engineer 评估):
   - tech-lead 与 backend-architect 在架构决策上重叠
   - code-reviewer 与 qa-engineer 在审查领域无清晰边界
   - context-manager 使用不可测量的触发阈值 ("10k tokens")
   - legal-advisor / ui-ux-designer 在方法论项目中几乎无触发场景

2. **开发阶段 Agent 自检不足** (QA 观察):
   - M3-slim B.2 开发中 3 个 Critical bug 未被开发 Agent 发现
   - 问题推迟到 pre-merge code-reviewer 才暴露
   - code-reviewer 的 description 缺少 B.2 阶段自动介入的触发锚点

3. **Agent 间无结构化协作契约** (AI Engineer 提议):
   - subagent-driver 调度各 Agent 独立工作,缺少 handoff protocol
   - 一个 Agent 的输出不会结构化传递给下一个 Agent

## Acceptance Criteria

1. 11 个 Agent 的 description 均符合 STCO 模式 (Scope-Trigger-Contract-Output, PromptX 三段式启发)
2. code-reviewer 与 qa-engineer 有明确的分工声明 (通过 STCO Trigger + Output 自然消歧)
3. Agent 间 handoff contract 定义完成 (含 agent_source: plugin|project 预留字段)
4. agent-router 路由置信度: 无 Agent 的 fallback 路由低于 0.80
5. Pilot 验证: api-documenter STCO 重写后 AB benchmark 通过

## Scope

- 修改: aria/agents/*.md (11 个文件)
- 修改: aria/skills/agent-router/ (路由规则增强)
- 不新增 Agent
- 不修改 Skill 逻辑 (仅 description + 路由)

## Decision Records

| ID | 决策 | 理由 |
|----|------|------|
| D1 | 质量优先于数量 — 不新增 Agent | 4/4 Agent Team 共识: 路由噪声风险 > 覆盖度收益 |
| D2 | Layer 1/2 拆分 — 本 US 仅含 description + 协作 | 产品负责人 "立即有用" + "避免短板" 双重目标分层实现 |
| D3 | Agent 保持无状态 (Fresh Subagent) | PromptX RoleX 进化模式与 subagent-driver 不兼容, 状态由 Skills 层管理 |
| D4 | STCO 模式 (非 Gherkin 语法) | LLM 路由基于语义匹配, 自然语言+结构化意图优于形式化语法 |
| D5 | Handoff contract 由 caller 注入 | description 用于路由, handoff 用于执行, 职责分离 |
