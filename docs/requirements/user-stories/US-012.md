## User Story

**作为** Aria 方法论的使用者 (同时维护 Forgejo 主仓库 + GitHub 镜像, 含子模块的项目)，
**我希望** Aria skills 自动检测并强制验证多远程之间的 HEAD 一致性，
**以便** 不再因单个远程推送遗漏导致插件市场版本滞后、子模块漂移、或团队协作偏差。

## 背景

### 事件回溯 (2026-04-12)

v1.14.0 发版时，aria 子模块的 2 个 commit (`06a0edf` release + `19f2861` PR #12 merge) 未被推送到 GitHub，直到人工验证才发现。根因分析:

1. **根本原因**: 并行 Bash 命令中，预期推送 aria 子模块的命令缺少 `cd /home/dev/Aria/aria` 前缀，实际在主仓库执行
2. **放大因素**: `git push` 在已同步仓库上输出 `"Everything up-to-date"` — 此信号在多仓库并行场景下语义歧义
3. **缺口**: 现有 state-scanner Phase 1.12 仅检查单个远程 (origin)，未验证主仓库和子模块 HEAD 在**所有**配置的远程上是否一致
4. **放大影响**: Claude Code 插件市场从 GitHub 拉取，GitHub 滞后 = 用户获取不到最新版本

**类似的历史事故** (CLAUDE.md 记录):
- 2026-04-10: aria v1.11.1 发版后未推送 GitHub, 市场停留在 v1.11.0

### 问题抽象

```
场景: N 个远程 + M 个仓库 (主 + 子模块)
需求: 所有 N*M 组合的 HEAD 一致
当前能力: 仅检测 1 个远程对 M 个仓库的同步 (缺 N-1 个远程的覆盖)
```

## 验收标准

### Layer 1: 被动检测 (state-scanner Phase 1.12 扩展)

- [ ] `sync_status` 输出扩展 `remotes[]` 数组，每个远程独立展示 HEAD SHA 和 parity
- [ ] 对主仓库 + 每个子模块，遍历所有 `git remote` 配置的远程
- [ ] 任一远程与本地 HEAD 不一致时报告 `multi_remote_drift` (warning 级)
- [ ] 单远程场景退化为现有行为，不触发新 warning
- [ ] 远程不可达 (auth 失败/仓库删除) 输出 `reachable: false + reason`，不静默为 `unavailable`
- [ ] 保留 `submodules[]` 原字段向后兼容 (指向 origin 的结果)
- [ ] D8 约束: 在 Phase 1.12 内原地扩展，不消耗新阶段配额

### Layer 2: 主动执行 (phase-c-integrator 强化)

- [ ] Phase C.2 合并后主动推送所有配置的远程 (非仅 origin)
- [ ] 推送后立即执行 SHA 验证: `git rev-parse HEAD <remote>/master` 必须等于本地 HEAD
- [ ] SHA 验证支持指数退避重试 (应对 Forgejo/GitHub 10-30s 复制延迟)
- [ ] 任一远程验证失败 → 阻断后续步骤，输出具体失败的 remote 名和修复命令
- [ ] `"Everything up-to-date"` 不再被接受为成功信号，必须有 SHA 一致性证据

### Layer 3: 共享基础设施

- [ ] `git-remote-helper` internal skill 可被 Layer 1 / Layer 2 引用 (实现细节见 Spec B)
- [ ] 定义 canonical JSON schema (所有消费方共用)
- [ ] 实现细节 (脚本语言、重试策略、跨平台兼容等) 见 [phase-c-integrator-push-enforcement](../../openspec/changes/phase-c-integrator-push-enforcement/proposal.md) §1

### 配置集成

- [ ] `.aria/config.json` 新增 `multi_remote` 配置块:
  - `enforced_remotes: []` (为空则自动发现所有 remote)
  - `fail_on_partial_push: true` (控制 Layer 2 是否阻断)
  - `read_only_remotes: []` (只读镜像白名单，push 失败降级为 warning)

## 技术备注

- **命令策略**: 日常 parity check 用本地 refs (快), 发版前 verify 用 `git ls-remote` (准确)
- **超时**: per-remote 5s 超时 + 整体 fail-soft, 失败降级为 `parity: unknown` 不阻断扫描
- **helper 命名**: `git-remote-helper` (遵循 git 生态命名惯例, 优于场景导向的 `multi-remote-helper`)
- **Phase 1.12 原地扩展**: 保留 D8 配额 (当前 14/15, 剩 1 格给未来真正新维度检测)
- **向后兼容**: 新字段 `remotes[]` 为可选, 消费方 null-check 即兼容
