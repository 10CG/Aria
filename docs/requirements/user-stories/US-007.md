# US-007: aria-orchestrator — 外部编排器与自动开发闭环

> **Priority**: HIGH
> **Status**: ready
> **Parent PRD**: prd-aria-v1.md
> **Target Version**: v1.5.0 (Phase 3b), v2.0.0 (Phase 3c)

---

## User Story

**作为** 使用 Aria 方法论的团队，
**我希望** 有一个独立的编排器服务，能监听 Issue 变化、定时触发心跳扫描、通过 `claude -p` 调用 Claude Code 执行十步循环，并在人类审批后合并 PR，
**以便** 实现从 Issue 提交到代码交付的半自动闭环，同时保留人类在关键节点的确认权。

## 背景

经 Agent Team 六轮讨论 (2026-04-03) 达成共识:

**Round 1-3**: 确认外部编排器方向
- Claude Code 会话是短暂的执行器，不适合做持久编排
- 编排逻辑 (状态机、队列、熔断) 应在外部独立进程中
- 通过 `claude -p` 调用保持对全部 33 Skills + 11 Agents 的复用

**Round 6**: 确认基于 Hermes Agent (NousResearch) 构建 (4:1)
- Hermes Agent 覆盖 80% 基础设施需求 (cron/gateway/memory/隔离执行)
- 模式 A: Hermes (Haiku) 做编排调度 → claude -p (Opus + Aria Skills) 做开发执行
- 生态关系: Aria 扩展 Hermes (不绑定，可替换运行时底座)
- 方法论可移植性本身是研究成果

## Aria 三层架构

```
方法论层 (standards/)           ← 定义如何思考/协作/决策
工具层   (aria/ plugin)         ← 33 Skills + 11 Agents, 人类交互式
运行时层 (aria-orchestrator/)   ← 系统自主编排, claude -p 执行
```

## 技术备注

- **运行时基座**: Hermes Agent (NousResearch, pip install hermes-agent, pin 版本)
- **双 LLM 层**: Hermes (GLM-4.7-Flash 免费, 做调度) → claude -p (Opus + 33 Aria Skills 做执行)
- **执行引擎**: Hermes terminal tool → `claude -p` (复用全部 Skill 生态)
- **状态存储**: Hermes FTS5 memory (替代自建 SQLite)
- **调度**: Hermes cron scheduler (tick/60s, 每 job 隔离 session)
- **人类审批**: Hermes 消息网关 (Telegram/Discord/Slack)
- **触发**: Forgejo webhook → Hermes gateway + cron 兜底
- **部署**: Aether Nomad service job (Docker backend)
- **Dashboard**: 编排器触发 `claude -p "/aria:dashboard"` 刷新，不重写看板逻辑
- **scan.sh**: 保持独立确定性脚本，Hermes 通过 terminal tool 调用
- **回退方案**: 若 Hermes 不满足需求，回退到 FastAPI+SQLite (~400 行 Python)
