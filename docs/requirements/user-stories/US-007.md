# US-007: aria-orchestrator — 外部编排器与自动开发闭环

> **Priority**: HIGH
> **Status**: ready
> **Parent PRD**: prd-aria-v1.md
> **Target Version**: v1.5.0 (Phase 3b), v2.0.0 (Phase 3c)

---

## User Story

**作为** 使用 Aria 方法论的团队，
**我希望** 有一个独立的编排器服务，能监听 Issue 变化、定时触发心跳扫描、通过 `claude -p` 调用 Claude Code 执行十步循环，并在人类审批后合并 PR，
**以便** 实现从 Issue 提交到代码交付的半自动闭环，同时保留人类在关键节点的确认权。

## 背景

经 Agent Team 三轮讨论 (2026-04-03) 达成共识:
- Claude Code 会话是短暂的执行器，不适合做持久编排
- 编排逻辑 (状态机、队列、熔断) 应在外部独立进程中
- 通过 `claude -p` 调用保持对全部 33 Skills + 11 Agents 的复用
- 编排器是 Aria 三层架构的"运行时层"，独立项目 `aria-orchestrator`

## Aria 三层架构

```
方法论层 (standards/)           ← 定义如何思考/协作/决策
工具层   (aria/ plugin)         ← 33 Skills + 11 Agents, 人类交互式
运行时层 (aria-orchestrator/)   ← 系统自主编排, claude -p 执行
```

## 验收标准

### Phase 3b: 编排器 MVP + 定时心跳 (v1.5.0)
- [ ] 独立项目 `aria-orchestrator` 仓库创建
- [ ] Aether Nomad service job 部署
- [ ] cron 定时触发 `claude -p` 执行心跳扫描
- [ ] SQLite 状态机: queued → analyzing → pr_created → resolved
- [ ] Forgejo webhook 接收 (HMAC-SHA256 签名验证)
- [ ] 熔断: 连续失败 3 次后暂停 + 告警
- [ ] 执行日志: .aria/heartbeat-log/{date}.md

### Phase 3c: 半自动开发闭环 (v2.0.0)
- [ ] `claude -p` 执行完整十步循环 (单 issue → PR)
- [ ] 分级安全策略:
  - Level 1: 全自动生成 PR，人类一键合并
  - Level 2: 自动 Spec → 暂停等人类确认 → 自动开发 → PR 需 Code Review
  - Level 3: 仅生成分析报告，禁止自动执行
- [ ] C.2 合并永远人类审批 (PR 是安全最后防线)
- [ ] git worktree 隔离执行
- [ ] 白名单标签触发 (aria:auto)
- [ ] 每 issue 最多重试 1 次，失败标 needs-human-review
- [ ] token 预算 + 每日执行上限

## 技术备注

- 技术栈: TypeScript (Claude Agent SDK 生态) 或 Python
- 执行引擎: `claude -p` (复用全部 Skill 生态，零重写)
- 状态存储: SQLite (进程重启后从最后成功 step 恢复)
- 触发: Forgejo webhook (issue.created) + cron 兜底
- 部署: Aether Nomad service job
- Dashboard 集成: 编排器触发 `claude -p "/aria:dashboard"` 刷新，不重写看板逻辑

## Decision Records

| D5 | Dashboard 不迁移到编排器 | 生成是工具，调度是运行时 | Tech Lead (Round 4, 5/5) |
