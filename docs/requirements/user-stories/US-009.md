## User Story

**作为** Aria 方法论的维护者 (以及使用者),
**我希望** 有一个透明度工具能区分 "内部 benchmark delta" 和 "跨项目 benchmark delta",
**以便** 在评估 Skill 质量时能看到它在 Aria-specific 约定外的真实价值, 但**不改变任何发版决策**.

## 背景

Forgejo Issue #8 调查 Aria 内部 AB 测试和 Kairos 跨项目测试的 delta 差距:

| Skill | Aria 内部 Δ | Kairos 外部 Δ | 差距 |
|-------|------------|--------------|------|
| state-scanner | +0.165 | +0.25 | +0.085 |
| commit-msg-generator | +0.80 | **0.0** | **-0.80** ⚠️ |

### 原 RCA 结论 (被 spike 证伪)

Phase 1-2 纸面 RCA 估算:
- state-scanner v2.9.0 有 ~50% 虚高
- commit-msg-generator 有 100% 虚高

计划 3 个 Level 3 Spec 重构 Rule #6 + Release Gate 2.0 + Escape Valve (~7-10 cycles).

### Spike 实测 (2026-04-10) 证伪

`aria-plugin-benchmarks/tools/calc_dual_delta.py` prototype 实测:
- state-scanner v2.9.0 真实 inflation **4.9%** (噪音级别)
- commit-msg-generator v2.0.1 真实 inflation **11.3%**

**关键 insight**: Kairos 测出的 0.0 不是内部造假, 而是 **eval scenario calibration 差异** (不同项目的 eval 测试场景难度不同, delta 差异是正常现象).

3 Level 3 Spec 降级为 **1 Level 2 Spec** 仅透明度增强.

## Meta-Lesson

本 US 验证了两个长期有效的 pattern:

1. **spike-first for data-driven hypotheses** — 数据驱动假说必须 spike 实测验证再立 Spec. 本 US 避免了 3 Level 3 Spec 和 ~1600 行无用工作.
2. **Level 2 patch 豁免 benchmark** — 纯文档/reporting 工具 patch, 不改 Skill 逻辑, 不需要跑 `/skill-creator benchmark` AB 对比.

两个 pattern 已沉淀到 MEMORY.md.
