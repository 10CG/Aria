> **我希望** 在 US-022 M2 Layer 1 状态机基础上, 完成 Layer 2 真实容器路径 wire-up (parameterized job + alloc 状态轮询) + 引入 Anthropic 官方 provider 双路 + 补完 schema migration + reconciler crash recovery,
> **以便** 让 ≥10 issue 走完整 cycle (S0→S9_CLOSE), 验证 M2 OD-11 carryover 验收 D (perf vs M1 × 1.5), 并准备 M4 human-gate 接入的稳定基座.

## Scope 摘要

将 US-022 M2 Layer 1 自动化"状态机驱动 dispatch"升级为"完整 Layer 1+Layer 2 闭环 + 双 provider + 重启可恢复":

**M2 carryover (硬性收尾, ~21h, per OD-11)**:
1. `aria-layer2-runner` parameterized job HCL — 解锁 S4_LAUNCH 后续 (S5_AWAIT polling, S6_REVIEW LLM, S7-S8 Forgejo)
2. `alloc_provider` production HTTP class — 30s ack-poll budget 真实化 + alloc-state 实时获取
3. Schema migration v2 — drop `schema.sql:98` inline UNIQUE constraint (替代 application-layer dedupe; 支持 `attempt_count > 1` retry)
4. 完整 cycle E2E + perf vs M1 (验收 D 真实 cycle wall-clock vs `m1_demo_002_p50 × 1.5`)

**M3 新增 (per OD-12, supersedes PRD §M3 90h with ×2.06 inflation 后真实 scope)**:
5. **GLM 多模型 routing (Q1=D')**: 在 Luxeno primary + Zhipu 官方 fallback 两 provider 内, 状态-感知 model 选择 (S2=4.5-air / S3=5-turbo / S6=5.1, fallback ladder 5.1→5-turbo→4.5-air); ProviderRouter 抽象 + ZhipuClient (stdlib urllib mirror SilknodeClient); fallback_chain_json dict-based (`[{provider, model, outcome, reason, latency_ms, ts}]`)
6. **Mechanical Reconciler (Q3=A)**: 独立 Nomad periodic job (`aria-layer1-reconcile.nomad.hcl`, 30min cadence) + CAS (compare-and-swap) 锁协议; 三阈值 (`max_age_at_S5_AWAIT_minutes=60`, `max_total_attempts=3`, `stuck_alloc_states=('pending','queued')`); 混合 retry 策略 (1-2 次 attempt_count++, 第 3 次 S_FAIL(stuck)) + Feishu 通知; M5 LLM-decided 升级 Strategy interface 预留
7. **Crash recovery**: Hermes 进程 kill -9 / 优雅重启 → S5_AWAIT dispatch 自动 resume polling (alloc_id 持久, alloc_provider re-query 后续推进); 不依赖 in-memory state
8. **Nomad integration 加固**: parameterized job idempotency 守卫 (per OD-5a + AD-M1-7) + 跨 alloc 失败 retry semantics; alloc_provider 真实化 (HTTP GET /v1/allocation/{id})
9. **Secret rotation T13 mid Phase B.2 (Q8d)**: 一次性 rotate 全 5 keys (含新 ZHIPU_API_KEY), post-rotation 用 验收 B perf benchmark 顺便验证

**核心交付** (待 brainstorm 细化):
- 完整 ≥10 issue cycle E2E demo (S0→S9_CLOSE 或 triage S_FAIL)
- 双 provider 切换实测 (Anthropic primary + Luxeno fallback, 或反转)
- Reconciler 单元 + 集成测试
- Schema migration v2 backward-compatible (existing dispatches.db 不丢数据)
- Crash recovery test: Hermes 重启后 S5_AWAIT 状态自动恢复到 polling
- M3 handoff (`m3-handoff.yaml`) — additive-only on M2 schema, 含 perf vs M1 实测 + 双 provider routing 决策回写

**估算** (per OD-12 Q2=E, 2026-05-03 lock):
- M2 OD-11 carryover: 21h
- PRD §M3 base scope (reconciler / crash recovery / Nomad hardening): 70h
- S2/S3 prompt scaffolding (ai R1 OBJ-2): 18h
- Q1=D' multi-model routing + benchmark: 8h
- Q1=D' ProviderRouter + ZhipuClient: 10h
- Audit overhead (混合 Phase A.2 4-round + B.2 scope-bounded + D 4-round): 25h
- Owner action items × 4: 4h
- Spec drafting + Phase A.2 audit: 12h
- 10% buffer: 17h
- **Total**: ~185h hard baseline / 5-6 weeks (超 PRD §M3 4-week, 起 OD-13 if 超 6 weeks)

**M2 input 契约** (从 [m2-handoff.yaml schema v1.0](../../../aria-orchestrator/docs/m2-handoff.yaml) 消费):
- `state_machine.fail_reason_enum` (9 values) — M3 reconciler 必须复用同枚举 (additive-only per AD-M1-7 + AD-M2-9 错误路由 matrix)
