## User Story

**作为** Aria 2.0 的方法论实施者，
**我希望** 在投入 ~750h 的 v2.0 开发之前完成所有关键假设验证与架构裁决，
**以便** M1+ 的开发不会基于错误前提推进，并在 Hermes 路线 Spike 给出量化数据后由产品负责人作出 Go/No-Go 决议。

## 背景

### v2.0 投资规模与不确定性

PRD v2.0 (Approved 2026-04-11) 估算总工时 ~750h / 30 周。M0 (前置验证) 是减少沉没成本的必要闸门。

经 **brainstorm + 4 轮 Agent Team 收敛评审**, M0 工时从初版 ~80h 扩展至最终 **~96.5h**, 吸收了以下精细化:

- 合规查证扩展 (T1 1h → 4.5h, 含 GLM ToS / luxeno / 跨境 / Aether 归属)
- Spike 可证伪量化 (cloc 协议 / license 扫描 / rebase 实操)
- 容器安全最小化验证 (read-only rootfs + bind mount 组合)
- 治理机制补齐 (`m0-handoff.yaml` schema + 签字闸门)

### Spike 的特殊地位

PRD AD3 (Hermes fork 决策) Agent Team R3 投票 2:2 分歧, Spike 是 timeboxed 1 sprint 的决策闸门。Spike 结论直接影响 US-022 (M2 Layer 1 状态机) 的实现路径。见独立 Spec [aria-2.0-m0-spike-hermes](../../../openspec/changes/aria-2.0-m0-spike-hermes/)。

### R1 的降级 (brainstorm D3 + Round 1 Legal 审议)

PRD 原将 R1 列为 "M1 hard precondition (需 Anthropic 邮件确认)"。经 brainstorm 降级为 legal-advisor agent 内审, 但经 Agent Team Round 1-4 扩展为 4.5h 的 **受限范围 Legal Memo**, 覆盖 5 项:
1. License 兼容性 (含 fork upstream 依赖链传染)
2. GLM 5.1 ToS 自动化条款 + luxeno 代理授权
3. luxeno 数据流跨境合规
4. Aether 节点物理归属 + 第三方 DPA 可能性
5. Docker pull 事实分发风险

Legal Memo 必须明确: 覆盖的司法辖区 / 未覆盖区域 / 风险提示 vs 合规结论的边界 / 失效条件。模板归档 `standards/legal/scoped-memo-template.md` (新增)。

## 验收标准 (PRD M0 Exit Criteria + Agent Team 收敛补齐)

### A. 架构决策文档收敛

- [ ] AD1-AD12 (PRD 列出的 12 项关键决策) 完整文档化到 `aria-orchestrator/docs/architecture-decisions.md`
- [ ] 每项 AD 含 alternatives considered 段落
- [ ] **T5 前置子任务**: Day 1 先产出 "AD1-AD12 索引草表" 交产品负责人锁定命名, 避免后期扯皮
- [ ] AD3 (Hermes fork 路径) 段落最终内容由 Spike 结论回填, 含两种路径的量化对比 (fork rebase 实测工时 + 自研 LoC)
- [ ] R7 (Nomad meta 64KB 限制) 作为 **M1 Layer 1→2 协议硬约定** 写入 AD 文档 (prompt 走文件, meta 只传 ISSUE_ID)
- [ ] **范围禁止**: M0 不创建 `standards/autonomous/` 目录骨架; 不起草 PRD §关联文档中的其他 6 个架构 doc (aria-2.0-overview / state-machine / layer2-container / container-launcher / hermes-fork-ops / prompt-templates), 这些属 M1+

### B. Hermes Spike 量化交付

- [ ] Spike 代码可运行 (gateway stub + SQLite state 最小原型), 存放于 `aria-orchestrator/spikes/hermes-route/`
- [ ] Spike 最小原型必含 **5 状态 4 转换** 硬演示: `idle → dispatched → running → awaiting_human → resumed`, 覆盖正常路径 + 1 个 human-in-loop
- [ ] Spike 元知识 prompt v0.1 (~1K token) 起草 + 在 Spike gateway 实跑验证 token 预算和格式
- [ ] **license 兼容性扫描** 作为 Spike 第 5 评估维度 (使用 `license-checker --production --json` + `pip-licenses --format=json --with-system`), 深度 transitive
- [ ] Spike 必须 **实际执行一次 rebase** (pull upstream / 解冲突 / 跑测试), 非纯 changelog 估算
- [ ] **量化交付指标**:
  - 自研路径 LoC (cloc 协议: `cloc --include-lang=Python --exclude-dir=.venv,__pycache__,migrations`, 测试 LoC 单列)
  - 自研路径开发工时 (实测)
  - fork 路径单次 rebase 工时 (实测) × ×1.5 安全系数 = 月度估算
  - upstream commit velocity 数据 (近 6 月, `git log --since="6 months ago" --oneline upstream/main | wc -l ÷ 6`)
  - direct + transitive license 全表 (GPL/AGPL/LGPL/Unknown 单列高亮)
- [ ] **Spike Report** 存放于 `openspec/changes/aria-2.0-m0-spike-hermes/spike-report.md` (Spec 交付物主位置), AD3 回填时以 OpenSpec 路径作为主引用
- [ ] **裁决规则**:
  - fork acceptable: 月度 rebase ≤ 20h (× 1.5 系数) 且 license 扫描无 GPL/AGPL/unknown 阻断
  - 自研 acceptable: cloc 业务 Python LoC ≤ 1000 且开发工时 ≤ 60h
  - 双 fail → R8 降级路径 (CLI-only)
  - 双 acceptable → 默认 fork, PRD patch PR 由产品负责人起草
  - 单边 acceptable → 选可行路径
- [ ] 若 Spike 结论与 PRD AD3 矛盾, 提交产品负责人**二次裁决**, 本 Story 在裁决前不可关闭

### C. 关键技术假设验证

- [ ] **A1 验证** (T3): `aria-runner:claude-latest` Dockerfile 初版构建成功
  - `claude -p` 在容器内能加载 `/root/.claude/plugins/aria` 并触发 state-scanner Skill
  - **`--read-only --tmpfs /tmp` 模式**下 headless plugin 加载不报错 (容器安全 P0 最小实证)
  - `/opt/aria-outputs/` bind mount 在 read-only rootfs 下仍可写 (read-only 与 bind mount 共存组合锁定)
  - Dockerfile 末部加构建时门控: `ARG DEPLOY_ENV=internal; RUN [ "$DEPLOY_ENV" = "internal" ] || exit 1`
  - README.md 头部并行维护 WARNING block (日期 + Dockerfile commit SHA)

- [ ] **A3 验证** (T2): Aether heavy-80/81/82 节点 NFS 挂载现状调查
  - `nomad alloc exec` 确认容器内 NFS 挂载点可见
  - parameterized job 实际写入 `/opt/aria-outputs/<file>` 后宿主读回
  - **宿主可见硬定义**: Nomad agent user 执行 `md5sum /opt/aria-outputs/<file>` 与容器内 md5 一致, 缺一即 fail

- [ ] **GLM smoke test** (T3 顺带, 非 A2 完整验证):
  - 用 **模板生成** (`aria-plugin-benchmarks/ab-suite/glm-smoke/templates/*.j2.md`) 的 5 条合成 prompt 跑 state-scanner
  - 通过条件 (二值化): ≥4 条触发 state-scanner 且返回非空 YAML (grep `current_phase:`)
  - 失败 → 升级 R8 评估 (GLM 完全不可用极端场景)
  - **所有 5 条样本** (pass/fail) 归档到 `aria-plugin-benchmarks/ab-suite/glm-smoke/failed-samples/<timestamp>-<seq>.yaml`, schema 含 `{prompt, failure_mode, raw_output, expected_grep, glm_model_version, status: pass|fail}`
  - `failure_mode` 5 值枚举: `not_triggered | triggered_empty_yaml | triggered_invalid_yaml | partial_response | timeout`
  - 模板禁止从真实仓库采样, 禁止匿名化真实代码; 模板须经 ai-engineer + **legal-advisor (人类) 最终签字** 后写入 `templates/REVIEW.md`

- [ ] **R1 Legal Memo** (T1):
  - 受限范围 5 项 (见 "背景" 段)
  - 产出 `aria-orchestrator/docs/r1-legal-memo.md`, 使用 `standards/legal/scoped-memo-template.md` 模板
  - **T1 前置闸门**: T1 完成前 T3/T4 不得启动 (T2 可启动, NFS 与合规无关)
  - T1 超时 (>5 工作日) 或升级邮件确认 → T3/T4 hold; T1 最终 No-Go → 进入 R8 评估
  - **R9 风险** (新增): GLM 5.1 ToS 自动化条款未正式确认, M0 Report 必须含 `## R9 官方回应状态` 三态章节 (`received / pending / declined`), M1 启动前置见 §D
  - **R10 风险** (条件新增): 若 Aether 节点非 10CG 自有且无 DPA → 升级 R10, 同款签字流程

### D. 产品负责人裁决与 Handoff

- [ ] **M0 Report** 撰写完成 (`aria-orchestrator/docs/m0-report.md`), 含: 架构决策 (含 Spike 结论) + 风险确认 + M1 精确路径 + R9 三态章节
- [ ] **`m0-handoff.yaml` schema v1.1** 作为 M0 Report 机读附件, 锁定字段集:
  ```yaml
  go_decision: Go | No-Go
  prd_patch_pr: <PR号> | "N/A"   # US-021 启动硬前置
  r9_status: received | pending | declined
  r9_signoff:                     # 仅 pending 时必填
    signed_by: human:<name>
    date: <YYYY-MM-DD>
    statement: "..."
  pd_signoff:                     # 产品负责人裁决签字
    signed_by: human:<name>
    date: <YYYY-MM-DD>
  spike_code_path: aria-orchestrator/spikes/hermes-route/
  nfs_path: /opt/aria-outputs/
  image_tag: aria-runner:claude-<sha>
  image_sha256: sha256:...        # R3-D8 构建产出, pin 下游
  glm_smoke_passed: true | false
  ad3_conclusion: fork | self-built | hybrid | deferred
  ```
- [ ] **签字统一表述**: 所有签字位 footer 必须包含: *"本签字仅作为决策可追溯记录 (audit trail), 不构成法律豁免、责任转移或正式法律意见。"*
- [ ] **Go/No-Go 机制** (维持 PRD 二元, 不引入三态):
  - **Go** → 进入 US-021, 若需 PRD 局部修订则同时起草 `prd_patch_pr`; PRD patch PR 必须 3 工作日内合并, 否则冻结 US-021 启动
  - **No-Go** → 触发 R8 退出策略 (CLI-only 降级模式) 或 PRD 整体重写
- [ ] **裁决机制** (US-020 内部, 不同步 standards/): 技术分歧出现时, orchestrator 输出 *推荐方案 + 备选 + 风险对比*, 进入 24h 产品负责人确认窗口; 超时未确认按推荐方案执行, 记录 `silent-approval` 到 decision log
- [ ] **US-021 task-planner 启动前置** (硬门控): `prd_patch_pr != null && (prd_patch_pr == "N/A" || forgejo GET /repos/10CG/Aria/pulls/{pr_num}.merged == true)`; 任一不满足则 hard block
- [ ] R9 `pending` 且无人类签字 → **audit-engine pre_merge 阶段 hard block** (不允许隐性推进)
- [ ] 产品负责人上调 velocity 系数 > ×2.0 时, 须附 velocity 数据快照 + **tech-lead 作为第二签字人** co-sign
- [ ] T1 No-Go 下 T2 产出 (NFS 实测) 部分有效 (基础设施事实, 不依赖合规), **knowledge-manager 在 M0 Report 中对 T2 涉及 GLM/luxeno 的段落执行 strikethrough 标注** (注明 "T1 No-Go 后失效, 日期:{date}"), **phase-c-integrator 在 pre_merge 检查点验证 strikethrough 已完成**, 失败则 block PR

## 风险与缓解

| ID | 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|---|
| **A1** | headless plugin 加载失败 | 中 | **极高** | v2.0 核心前提颠覆; 触发 R8 退出策略评估 |
| **A3** | NFS 不可用 | 低 | 高 | 单节点 constraint pin |
| **A5** | Hermes fork rebase 超预期 | 中 | 高 | timeboxed Spike (T4 ≤60h 评估, >72h 强制终止) |
| **R1** | Legal 内审发现新疑点 | 低 | 中 | 升级 ad-hoc 法律咨询 (体外预算); T1 hold |
| **R9** | GLM ToS 官方不回应 | 中 | 中 | M1 启动前须显式签字 (pending 态), audit-engine hard block |
| **R10** | Aether 节点第三方归属 | 未知 | 中 | T1 +0.5h 查证, 若非自有 → DPA 或签字流程 |
| **A2/smoke** | GLM 完全不可用 | 低 | 高 | T3 二值化 smoke test; 失败 → R8 评估 |
