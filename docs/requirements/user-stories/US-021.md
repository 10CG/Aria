## User Story

**作为** Aria 2.0 的方法论实施者，
**我希望** 在 M0 给出的架构 blueprint (aria-runner 镜像 / Nomad parameterized dispatch / host volume) 之上实现一个端到端可运行的手动 MVP: 从 DEMO issue 输入 → Nomad 分派 → aria-runner 容器执行 → Forgejo PR 产出，
**以便** 证明 Aria 2.0 Layer 2 容器架构在生产路径 (registry push + cross-node dispatch + PR 回写) 可运行，为 M2 Layer 1 状态机集成提供稳定基座，并把 Layer 1/Layer 2 的开发风险解耦。

## 背景

### M0 已交付的 blueprint

M0 (US-020) 已把以下假设实证并固化:

| 项 | M0 产出 | M1 消费 |
|----|---------|--------|
| A1 headless plugin | `aria-runner:claude-m0` (29 skills loaded / 5/5 hit) | 升级为 `claude-latest` + registry push |
| A3 shared storage | virtiofs 透传 (9/9 md5 cross-node identical) | 复用 `/opt/aether-volumes/aria-runner/outputs` host volume |
| R7 Nomad meta boundary | ~128 KiB (Linux kernel, 非 64 KiB) | Layer 1→2 协议仅传 ISSUE_ID, prompt 走 bind mount |
| AD3 | Option C Extension-only (hermes 仅 tool pack, 0 fork) | M1 不触及 (Layer 1 延后 US-022); **仅固化 Layer 2 行为契约** |
| R8 | virtiofs instead of NFS (spec override) | Nomad job stanza 复用 M0 实测 HCL |
| GLM smoke (T3.4) | Luxeno Anthropic endpoint 40% timeout 基线 | **M1 走 Anthropic 官方 API**, 不依赖 Luxeno; Luxeno 路径延后到 M2 |

### M1 为什么是 "手动 MVP" 而非完整自动化

PRD v2.1 §M1 明确 out-of-scope: Hermes Layer 1 集成 / SQLite 状态机 / 自动触发。选择手动 MVP 的理由:

1. **解耦 Layer 1/2 开发风险**: M2 Hermes 集成估 140h, 与 M1 100h 并列为 v2.0 最大两块工时。先把 Layer 2 跑通, 可在 Layer 1 开发时用稳定的 Layer 2 做集成测试。
2. **验证 Layer 2 生产性假设**: M0 Spike 是 `docker run` 单机, M1 要证明 registry push + cross-node Nomad dispatch + Forgejo PR 闭环可工作 — 这是 M0 没有覆盖的三个新维度。
3. **降低"Layer 1 推动不了"的系统性风险**: 若 M1 MVP 失败 (如 registry 认证 / Nomad volume / Claude CLI 不稳定), 则 M2 的 140h 需要重估, 决策点前移 2 个月。

### Blueprint Coupling

M0 Report §6.1-6.2 已给出 M1 具体 HCL template 草稿。本 Story 把草稿转化为可执行产物:
- Nomad job file (从草稿 → 实际运行)
- 镜像 tag 策略 (`claude-m0` → `claude-<sha>` or `claude-latest`)
- Registry push 流程 (Forgejo container registry 认证 + push script)
- 手动 dispatch 脚本 + DEMO issue 输入协议

## 验收标准 (from PRD v2.1 §M1)

### A. 端到端 MVP 可运行

```
Given: 1 个预先准备好的 DEMO issue (.aria/issues/DEMO-001.yaml)

Do:
  1. 本地 docker build -t forgejo.10cg.pub/10CG/aria-runner:claude-latest
  2. docker push forgejo.10cg.pub/10CG/aria-runner:claude-latest
  3. nomad job run aria-runner-template.hcl (parameterized)
  4. nomad job dispatch -meta ISSUE_ID=DEMO-001 aria-runner-template
  5. 手动 git worktree + 手动拷 inputs 到 /opt/aria-inputs/DEMO-001/

Verify:
  - Nomad alloc complete (non-zero duration, exit code 0)
  - /opt/aria-outputs/DEMO-001/result.json 生成且 schema valid
  - Forgejo 出现新 branch (naming convention: aria/DEMO-001)
  - Forgejo 出现 PR, diff 非空, diff 内容符合 issue 描述
```

### B. 容器镜像生产化

- `aria-runner:claude-<sha>` + `aria-runner:claude-latest` 双 tag 推送成功
- 镜像 SHA256 记录到 M1 handoff (对齐 M0 `image_sha256` 字段模式)
- Registry auth 文档化 (`.env.local` 模式, 不入 git)
- Dockerfile 构建时门控 (`DEPLOY_ENV=internal` 检查) 保留

### C. Issue 输入协议 v0.1

- DEMO issue schema 定义 (YAML):
  - `id`, `title`, `description`, `files[]` (输入文件清单), `expected_changes` (验收描述)
- 输入拷贝脚本 (bash, 幂等): `.aria/issues/{ID}.yaml` → `/opt/aria-inputs/{ID}/`
- **可 breaking**: M2 Hermes 集成时允许 schema v0.1 → v1.0 breaking change (避免本 MVP 过早锁死)

### D. Runner 执行协议

- Nomad alloc 启动 → 读 ISSUE_ID → load `/opt/aria-inputs/{ISSUE_ID}/` → 调用 `claude -p --permission-mode bypassPermissions`
- Claude 执行产出 commit → push 分支 → 创建 PR → 写 `/opt/aria-outputs/{ISSUE_ID}/result.json`
- 失败兜底: Claude timeout / git conflict / PR 创建失败 → result.json 记录 error + reason, alloc exit non-zero

### E. M1 Report + handoff

- `aria-orchestrator/docs/m1-report.md` (简化版 M0 Report)
- `aria-orchestrator/docs/m1-handoff.yaml` (记录镜像 sha256 / registry URL / DEMO 执行结果)
- M2 (US-022) 启动硬门控: 本 Handoff 的 `e2e_demo_passed == true`

## 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| **R-M1-1** Forgejo registry push 认证失败 | Medium | High (阻断 MVP) | Spike-first: 第一周先验证 `docker push` 到 Forgejo, 失败则升级到 nomad artifact 方式 |
| **R-M1-2** Nomad parameterized dispatch 行为变化 | Low | High | 复用 M0 T2.2 已验证的 dispatch 路径, 不引入新 Nomad 版本 |
| **R-M1-3** Issue 输入协议过早锁死 | Medium | Medium | 显式标记 v0.1 breaking change 允许, M2 重塑时不背 compat 债 |
| **R-M1-4** Claude -p 在 Nomad alloc 内行为异常 (M0 只单机测过) | Medium | High | T4 早期 spike 在 alloc 内跑一次 claude 调用, 失败则暴露 |
| **R-M1-5** Forgejo API create PR 权限问题 | Low | Medium | 预先创建 bot 账户 + token, alloc 内用 bot 身份 |
| **R-M1-6** DEMO issue 设计反复 | Medium | Low | DEMO schema v0.1 显式允许 breaking (escape hatch 覆盖); DEMO 全部 synthetic (自建 fixture repo `aria-plugin-benchmarks/ab-suite/m1-mvp/fixtures/`), 避免 real IP 反复锁定 |
| **R-M1-7** 工时估算偏差 (PRD 100h 基线) | **Medium** | Medium | M0 velocity ×3.76 为特例不迁移; M1 保留 20h buffer; **Week 2 checkpoint**: 累计工时 ≥60h 且 T4 未开始 → 触发 scope 裁剪 (DEMO-002 降级为 optional) 并写入 handoff.open_issues; **降级语义 (per TL-M3 / AD-M1-5)**: DEMO-002 降级时 `e2e_demo_passed = DEMO-001.passed AND DEMO-002.status ∈ {passed, deferred-with-approval}`, deferred 需 owner 书面签字 + 自动走 No-Go-with-revision 分支 (非 Go) |
| **R-M1-8** Legal carryover: Anthropic 官方 API 跨境合规假设 | **Medium** | **High** | T1 交付 `m1-legal-carryover.md`: M0 IS-4 结论失效审视 + Anthropic ToS §7-8 逐条核查 + R9 GLM legacy / R10 Aether 节点归属复查; `anthropic_api_terms_verified=false` 阻断 Go decision (per AD-M1-6 / LA-C1) |
