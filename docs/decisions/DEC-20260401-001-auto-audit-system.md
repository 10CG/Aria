# 决策: DEC-20260401-001 - 十步循环自动审计系统设计

> **日期**: 2026-04-01 | **模式**: technical

## 背景

当前 agent-team-audit 为单轮审计，存在深度不足、无对抗机制、检查点有限、无收敛保证等问题。需要设计多轮收敛的审计系统。

## 约束条件

| 类型 | 约束 | 影响 |
|------|------|------|
| 成本 | 多轮 x 多 Agent = token 消耗大 | 必须默认关闭，opt-in |
| 兼容 | 现有 agent-team-audit 用户不受影响 | 向后兼容 |
| 确定性 | Agent 分组不应依赖运行时判断 | 配置化 |

## 考虑的方案

| 决策点 | 选项 | 选择 | 理由 |
|--------|------|------|------|
| 审计模式 | 单轮 / 多轮收敛 / 多轮对抗 | 多轮收敛 + 多轮对抗两种 | 不同环节侧重不同（质量/决策/共识） |
| 配置策略 | 默认开启 / 默认关闭+adaptive | 默认关闭 + adaptive 可选 | 成本敏感，opt-in |
| 收敛判定 | 结论比较 / 投票 / 混合 | 混合（结论无变化 + 全票 PASS） | 双重验证更可靠 |
| 收敛保护 | 硬上限 / 预算 / 人工gate / 自适应 | 硬上限 max_rounds=5，可配置 | 简单可靠 |
| Agent 分组 | 固定 / 动态 / 轮换 | 按检查点动态分组（配置化） | 不同议题需要不同的挑战视角 |
| 检查点数量 | 3 已有 / +4 新增 | 7 个全覆盖 | 按复杂度 adaptive 控制哪些启用 |
| 配置生成 | 手动 / 新 Skill / config-loader 扩展 | config-loader 扩展 | 避免一次性 Skill，加载时自动处理 |
| 项目初始化 | 模板复制 / 自动分析生成 | 自动分析项目特征 → 裁剪生成 | 零配置体验 |

## 最终选择

两种审计模式 + 7 检查点 + adaptive 自适应 + config-loader 自动初始化。

## 理由

1. convergence 模式适合质量保障（实现审查、任务分解），challenge 模式适合决策验证（Spec、架构）
2. adaptive 模式让用户零配置即可获得合理的审计策略
3. 配置化分组确保行为确定性，不依赖 AI 运行时判断
4. max_rounds 硬上限防止成本失控

## 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| Token 消耗过高 | 默认关闭 + adaptive 按复杂度选择 + max_rounds |
| 收敛时间过长 | max_rounds=5 硬上限 + 未收敛时人工介入 |
| Agent 分组不合理 | 预配置默认值 + 用户可覆盖 + config-loader 按项目裁剪 |
| 与现有 agent-team-audit 冲突 | 向后兼容，audit.enabled=false 时现有行为不变 |
