## 核心设计决策

### 1. 三层模式架构

```
Level 1: problem (问题空间探索)
    ↓
Level 2: requirements (需求分解)
    ↓
Level 3: technical (技术方案设计)
```

**设计理由**:
- 需求澄清在不同阶段有不同的关注点
- 分层设计使每个阶段可以独立使用
- 与 AI-DDD 方法论的自然分层对齐

### 2. 状态机驱动的对话流程

```
INIT → CLARIFY → EXPLORE → CONVERGE → SUMMARY → COMPLETE
```

**设计理由**:
- 标准化的对话流程确保质量
- 状态可追踪，便于调试和优化
- 支持中途退出和恢复

### 3. 决策优先于文档

```
对话 → 决策记录 → 文档生成
```

**设计理由**:
- AI-DDD 的核心是"协作思考"
- 决策记录比文档更有价值（记录"为什么"）
- 文档可以由决策自动生成

### 4. 约束驱动的方案过滤

```
约束收集 → 方案过滤 → 可行方案对比
```

**设计理由**:
- 硬约束快速排除不可行方案
- 软约束用于方案评分
- 约束变更时自动重新评估

## 关键特性

| 特性 | 描述 | 优势 |
|------|------|------|
| **智能引导** | 根据对话状态动态生成问题 | 避免固化流程，适应不同场景 |
| **深度控制** | 监测模糊度、覆盖面、共识度 | 防止过早收敛或无休止讨论 |
| **约束验证** | 自动检查方案是否满足约束 | 快速排除不可行方案 |
| **决策记录** | 结构化记录"为什么选A而非B" | 可追溯，可复审 |
| **多模式** | problem/requirements/technical | 覆盖需求形成全周期 |

## 与 Superpowers 的差异

| 维度 | Superpowers brainstorming | Aria brainstorm |
|------|--------------------------|-----------------|
| **定位** | 技术讨论工具 | AI-DDD 协作思想载体 |
| **时机** | 代码实现前 | 需求形成全周期 |
| **模式** | 单一模式 | 三种模式分层 |
| **输出** | 会话记忆 | 持久化决策日志 |
| **约束** | 无 | 完整的约束管理系统 |
| **集成** | 独立使用 | 深度融入方法论 |
| **领域建模** | 无 | 核心能力 |

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |

| 对话轮次过多 | 中 | 中 | 设置合理上限，支持提前退出 |

| 与现有流程冲突 | 低 | 中 | 保持向后兼容，可选启用 |
| 性能问题 | 低 | 低 | 增量处理，缓存优化 |
