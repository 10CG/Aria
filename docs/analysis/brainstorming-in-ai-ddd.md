# 头脑风暴在 AI-DDD 方法论中的定位与设计

> **分析日期**: 2026-02-05
> **基于**: Aria vs Superpowers 对比分析
> **目标**: 设计合理的 brainstorming skill 集成方案

---

## 一、核心问题：头脑风暴应该放在哪里？

### 1.1 方法论对比视角

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    需求澄清的三个层次                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Layer 1: 业务问题探索                                                   │
│           "我们要解决什么问题？"                                          │
│           ↓                                                              │
│  Layer 2: 需求分解与优先级                                               │
│           "具体需要哪些功能？"                                            │
│           ↓                                                              │
│  Layer 3: 技术方案设计                                                   │
│           "如何实现这些功能？"                                            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

| 方法论 | 主要关注点 | 头脑风暴位置 | 文档产出 |
|--------|-----------|-------------|----------|
| **Superpowers** | Layer 3 (实现层) | 代码实现前 | 无正式文档 |
| **Aria (当前)** | Layer 2-3 (需求+实现) | 无 | PRD + OpenSpec |
| **Aria (建议)** | Layer 1-2-3 (全链路) | 三个阶段 | 交互记录 + 文档 |

### 1.2 AI-DDD 的核心假设

```
传统模式: 人类思考 → 输出文档 → AI 读取理解
Aria 模式: 人机协作思考 → 共同理解 → 输出文档

关键差异:
  - 头脑风暴是"共同思考"的过程
  - 文档是"思考成果"的固化
  - 没有"思考过程"的文档是无意义的
```

**结论**: 头脑风暴不应该是一个可选的附加功能，而应该是 AI-DDD 方法论的**核心环节**。

## 二、设计原则：Aria 风格的头脑风暴

### 2.1 与 Superpowers 的关键差异

| 维度 | Superpowers brainstorming | Aria 设计建议 |
|------|--------------------------|---------------|
| **触发时机** | 代码实现前 | 需求形成全周期 |
| **输出** | 设计决策 | 设计决策 + OpenSpec 草稿 |
| **记录** | 会话记忆 | 持久化决策日志 |
| **约束管理** | 代码级约束 | 业务 + 技术 + 架构约束 |
| **领域建模** | 无 | 核心关注点 |

### 2.2 Aria 特有的核心需求

```yaml
AI-DDD 头脑风暴的核心:
  1. 领域理解
     - 业务术语统一定义
     - 领域边界识别
     - 核心领域概念建模

  2. 约束收集
     - 业务约束 (预算、时间、合规)
     - 技术约束 (架构、依赖、兼容性)
     - 团队约束 (技能、偏好)

  3. 决策记录
     - 为什么选择方案A而非B
     - 决策的假设条件
     - 决策的可撤销性评估
```

## 四、Skill 设计：brainstorm 核心

### 4.1 Skill 结构

```yaml
name: brainstorm
description: |
  AI-DDD 头脑风暴引擎，支持问题空间探索、需求分解、技术方案设计。
  通过多轮对话澄清需求，记录设计决策，生成结构化输出。

version: 1.0.0

modes:
  problem: 问题空间探索
  requirements: 需求分解
  technical: 技术方案设计

features:
  - 多轮对话引导
  - 约束条件管理
  - 决策点记录
  - 领域术语统一
  - 方案对比分析
```

### 4.2 核心能力

#### 能力 1: 智能引导对话

```yaml
对话引导策略:
  开场:
    - 识别用户输入的模糊程度
    - 选择合适的引导模板
    - 设定头脑风暴边界

  推进:
    - 检测讨论深度
    - 在适当时机提出尖锐问题
    - 防止过早收敛

  收敛:
    - 检测共识形成
    - 总结关键决策点
    - 生成下一步建议
```

#### 能力 2: 决策记录

```yaml
决策记录格式:
  decision:
    id: DEC-001
    date: 2026-02-05
    context: [背景描述]
    options:
      A: [方案A描述]
      B: [方案B描述]
    selected: A
    rationale: [选择理由]
    assumptions: [假设列表]
    reversible: true/false
    expiry: [决策有效期，如"v2.0前"]
```

#### 能力 3: 约束管理

```yaml
约束分类:
  business:
    - 预算限制
    - 时间限制
    - 合规要求

  technical:
    - 架构约束
    - 技术栈限制
    - 性能要求

  team:
    - 技能偏好
    - 资源可用性

约束验证:
  - 每个方案检查约束冲突
  - 标记不可行方案
  - 约束变更时重新验证
```

### 4.3 与现有 Skills 的集成

```yaml
集成点:

1. state-scanner
   输出: "发现模糊需求" → 触发 problem-brainstorm

2. spec-drafter
   输入: 来自 technical-brainstorm 的决策记录
   输出: 预填充的 proposal.md

3. requirements-validator
   输入: 来自 requirements-brainstorm 的 User Stories
   验证: 决策一致性检查

4. task-planner
   输入: 来自 technical-brainstorm 的技术方案
   输出: 更精确的任务分解
```

## 六、决策记录格式

### 6.1 决策日志文件

```markdown
# 设计决策记录: AI 客服功能

> **Brainstorm ID**: BRAIN-2026-02-05-001
> **创建日期**: 2026-02-05
> **状态**: Active

## 问题定义

### 原始需求
"给应用加一个 AI 功能"

### 澄清后的问题
用户需要快速获取产品帮助，当前 FAQ 无法满足个性化问题。

### 业务目标
- 减少客服工作量 30%
- 提升问题解决率至 80%
- 保持 24/7 可用性

## 约束条件

| 类型 | 约束 | 影响 |
|------|------|------|
| 业务 | 预算 < $500/月 | 排除昂贵托管服务 |
| 技术 | 必须私有化部署 | 需要自建 RAG |
| 合规 | 数据不得出境 | 只能使用国内模型 |

## 考虑的方案

### 方案 A: OpenAI Assistants API
- **优点**: 快速上线，功能完整
- **缺点**: 数据出境，不满足合规要求
- **结论**: ❌ 不可行

### 方案 B: 自建 RAG (FAISS + 本地模型)
- **优点**: 私有化，成本低
- **缺点**: 需要维护，效果可能不如托管服务
- **结论**: ✅ 选中

### 方案 C: 混合方案 (托管 + 本地备份)
- **优点**: 灵活性高
- **缺点**: 复杂度高，成本中等
- **结论**: ⭐ 备选，方案 B 失败时考虑

## 技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 向量存储 | FAISS | 开源免费，满足初期需求 |
| 嵌入模型 | bge-m3 | 中文效果好，开源 |
| 生成模型 | Qwen2.5-7B | 本地部署可行 |
| 前端框架 | 复用现有 | 避免技术栈膨胀 |

## User Stories 分解

| ID | Story | 优先级 | 复杂度 |
|----|-------|--------|--------|
| US-001 | 基础问答功能 | P0 | Level 2 |
| US-002 | 产品文档接入 | P0 | Level 2 |
| US-003 | 反馈机制 | P1 | Level 1 |
| US-004 | 多轮对话 | P2 | Level 3 |
| US-005 | 答案来源引用 | P2 | Level 2 |

## 下一步

1. 创建 PRD: `prd-ai-customer-service-v1.md`
2. US-001 创建 OpenSpec
3. 技术验证: FAISS + bge-m3 POC

| 日期 | 变更 | 原因 |
|------|------|------|
| 2026-02-05 | 初始记录 | - |
```

## 九、结论

### 回答原问题

> "在 PRD 和 User Story 的创建阶段，是否就应该添加头脑风暴 skill？"

**答案**: 不仅是"应该"，而是**必须**。

**理由**:
1. **AI-DDD 的本质是协作思考**，没有头脑风暴就是人类单向输出
2. **PRD/User Story 是思考的产物**，没有思考过程的文档没有价值
3. **决策记录比文档更重要**，"为什么"比"是什么"更关键

### 最佳实践建议

```yaml
头脑风暴在 Aria 中的定位:
  1. 不是可选功能，而是核心环节
  2. 不是独立 skill，而是融入工作流
  3. 不是一次性对话，而是持续记录
  4. 不是技术讨论，而是业务 + 技术双轮驱动

实施策略:
  1. 先在 technical-brainstorm 验证效果
  2. 再扩展到 requirements-brainstorm
  3. 最后覆盖 problem-brainstorm
  4. 持续优化对话引导逻辑
```
